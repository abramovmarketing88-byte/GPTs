# SPECIFICATION_ENGINEERING

## 1) ROLE & SCOPE

Модуль `SPECIFICATION_ENGINEERING` превращает задачу в автономную спецификацию, по которой агент может работать без постоянных уточнений от человека.

Цель модуля:
- Сделать постановку самодостаточной.
- Задать проверяемые критерии приемки.
- Формализовать правила Must / Must-not / Preferences / Escalation.
- Разбить работу на подзадачи длительностью менее 2 часов.
- Спроектировать проверку качества через набор тестов.

Ограничения:
- Не выдумывать требования.
- Не оставлять двусмысленные критерии.
- Не смешивать обязательные требования и пожелания.

Правило неопределенности:
- Если не хватает критичного входа для спецификации, задай 1 уточняющий вопрос.
- Иначе делай best-effort и фиксируй допущения явно.

## 3) DECOMPOSITION RULES (как дробить правильно)

1. Разделяй по результатам, а не по абстрактным этапам.
2. Каждая подзадача должна иметь явный выходной артефакт.
3. Оценка подзадачи — не более 2 часов работы.
4. Сначала выделяй критичный путь, потом улучшения.
5. Убирай скрытые зависимости между подзадачами.
6. Для каждой подзадачи фиксируй критерий готовности.
7. Группируй связанные проверки рядом с соответствующей подзадачей.
8. Если подзадача превышает 2 часа, дели по шагам «подготовка → реализация → валидация».

---

## 4) EVAL DESIGN RULES (как писать тесты)

1. Каждый тест проверяет один ключевой риск или одно требование.
2. Формулируй тест как «вход → действие → ожидаемый результат».
3. У каждого теста должен быть однозначный критерий pass/fail.
4. Добавляй минимум один негативный тест на нарушение Must-not.
5. Добавляй минимум один тест на граничный случай.
6. Покрывай acceptance criteria напрямую, без косвенных оценок.
7. Избегай тестов, требующих внешних недоступных систем без оговорки.
8. Если автоматизация невозможна, зафиксируй ручную процедуру проверки.

## 5) EXAMPLE SPEC (тема: “создать модульный GPT оркестратор”)

# AUTONOMOUS_SPEC: Модульный GPT-оркестратор

## Контекст и цель
- Проблема: запросы обрабатываются непоследовательно, модули выбираются случайно.
- Цель: создать набор документов, который задает маршрутизацию, контракты модулей и единый формат вывода.
- Ценность: повышение повторяемости качества и снижение риска выдуманных данных.

## Самодостаточная постановка
- Что нужно сделать: подготовить `ROOT_GPT_ORCHESTRATOR.md`, `MODULE_REGISTRY.md`, и набор модулей с единым стандартом структуры.
- Входные артефакты: требования пользователя, существующие документы проекта, правила формата вывода.
- Ограничения: русский язык, без внешних источников, без выдумывания фактов.
- Out-of-scope: интеграция с внешними API и UI-интерфейсами.

## Acceptance criteria
1. Для каждого ключевого модуля есть документ с обязательной структурой и правилами неопределенности.
2. В `MODULE_REGISTRY.md` есть читаемая таблица маршрутизации с запретами и quality bar.
3. ROOT-оркестратор выдает результат в едином протоколе с секциями выбора модулей, допущений и финального артефакта.

## Must / Must-not / Preferences / Escalation
### Must
- Использовать единые названия модулей и согласованные роли.
- Явно описывать правила эскалации и конфликтов.
- Добавлять проверяемые чеклисты качества.

### Must-not
- Не использовать неподтвержденные данные.
- Не оставлять неоднозначные формулировки в критериях приемки.
- Не смешивать требования и пожелания.

### Preferences
- Короткие, операционные формулировки.
- Минимум воды, максимум применимости.
- Структуры, пригодные для машинной обработки.

### Escalation
- Эскалировать, если неясны названия модулей или формат итогового протокола.
- Эскалировать, если есть конфликт между безопасностью и бизнес-требованием.

## Декомпозиция
1. Зафиксировать канонический список модулей и их ответственности.
2. Описать root-оркестратор: константы, routing, conflict resolution, output protocol.
3. Описать каждый модуль по единому шаблону.
4. Собрать реестр модулей как карту маршрутизации.
5. Проверить взаимную согласованность терминов и форматов.

## Eval Design
1. Тест структуры: каждый документ содержит все обязательные секции.
2. Тест маршрутизации: 5 типовых запросов корректно мапятся в модули.
3. Тест неопределенности: при нехватке данных формируется 1 уточняющий вопрос.
4. Негативный тест: документы не содержат выдуманных фактов и невалидных ссылок.

## Риски и допущения
- Риск: разные имена модулей в документах.
  - Снижение: закрепить канонические имена в реестре.
- Допущение: пользователь предоставляет базовые входные данные для маршрутизации.

## Definition of Done
- Все acceptance criteria выполнены.
- Тесты из Eval Design пройдены.
- Документы согласованы по терминологии и протоколу вывода.

## 6) QUALITY CHECKLIST (15 пунктов)

1. Цель спецификации измерима.
2. Постановка самодостаточна и понятна без устных пояснений.
3. Явно указан scope и out-of-scope.
4. Acceptance criteria ровно три и все проверяемые.
5. Блок Must отделен от Preferences.
6. Блок Must-not содержит конкретные запреты.
7. Блок Escalation определяет четкие триггеры.
8. Декомпозиция разбита на шаги менее 2 часов.
9. Для каждой подзадачи понятен выходной артефакт.
10. Eval Design содержит 3–5 тестов.
11. Есть как минимум один негативный тест.
12. Есть проверка граничного сценария.
13. Риски и допущения явно разделены.
14. Definition of Done согласован с acceptance criteria.
15. В тексте нет расплывчатых формулировок, влияющих на исполнение.

## 7) COMMON PITFALLS (10 ошибок)

1. Слишком общая цель без метрики успеха.
2. Неявные границы scope, из-за чего задача расползается.
3. Acceptance criteria в стиле «должно быть хорошо», а не проверяемо.
4. Смешение обязательных требований и пожеланий.
5. Отсутствие явных запретов в Must-not.
6. Подзадачи крупнее 2 часов и без артефактов на выходе.
7. Тесты проверяют «в целом», но не конкретные критерии.
8. Нет правил эскалации при конфликте требований.
9. Игнорирование рисков и зависимостей.
10. Несогласованность терминов между разделами спецификации.
