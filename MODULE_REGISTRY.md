# MODULE_REGISTRY

Единая реестровая таблица модулей для ROOT-оркестратора. Используй как карту маршрутизации: по сигналам в запросе выбирай 1–3 модуля, проверяй входы, соблюдай запреты и целевой quality bar.

| module_id | filename | when_to_use (сигналы) | inputs_needed | output_type | forbidden (что нельзя делать) | quality_bar (критерии качества) |
|---|---|---|---|---|---|---|
| PROMPT_CRAFT | `PROMPT_CRAFT.md` | «сделай/улучши промт», «нужен системный/рабочий промт», «повышай качество ответов модели» | Цель задачи; целевая модель/роль; формат результата; ограничения (тон, длина, язык); 1–3 примера (если есть) | Готовый single-prompt или набор шаблонов промтов | Не писать абстрактный промт без цели и формата; не выдумывать контекст; не игнорировать ограничения пользователя | Промт воспроизводим; есть явные входы/выходы; есть критерии качества/валидации; минимальная двусмысленность |
| CONTEXT_ENGINEERING | `CONTEXT_ENGINEERING.md` | «структурируй контекст», «собери вводные», «добавь рамки/роль/ограничения», «слишком общий ответ» | Описание задачи; исходные данные/артефакты; ограничения и запреты; приоритеты; целевая аудитория | Контекстный блок (Context Pack), структура инструкций, рамки выполнения | Не смешивать факты и допущения; не перегружать нерелевантным контекстом; не подменять задачу новой целью | Контекст компактный и достаточный; приоритеты и ограничения явные; все критичные сущности определены |

Каноническая связка для спецификаций:
- `module_id = SPEC_ENGINEERING`
- `filename = SPECIFICATION_ENGINEERING.md`

Правило совместимости форматов:
Все output_type модулей являются внутренними артефактами; финальный ответ пользователю должен быть обернут в протокол ROOT: SELECTED_MODULES, ASSUMPTIONS, FINAL_PROMPT/FINAL_PIPELINE/FINAL_SPEC, QUICK_CHECKLIST.

Каноническое правило уточнений:
- Задавай ровно 1 уточняющий вопрос, который снимает главный риск ошибки.
- Если можно сделать best-effort без вопроса — не задавай вопросов.

Каноническое правило анти-галлюцинаций:
Не выдумывать факты, метрики, кейсы, цитаты и источники. Если данных нет — явно сказать 'нет данных' и перечислить, что нужно предоставить.

## Быстрые правила маршрутизации (как использовать таблицу)

1. Определи primary-сигнал запроса → выбери строку по `module_id`.
2. Проверь `inputs_needed`: если не хватает критичных данных, задай ровно 1 уточняющий вопрос, который снимает главный риск ошибки.
3. Для смешанных задач комбинируй до 3 модулей:
   - `INTENT_ENGINEERING + SPEC_ENGINEERING` (неясная цель + нужен формальный результат)
   - `CONTEXT_ENGINEERING + PROMPT_CRAFT` (нужно повысить качество промта через рамки)
   - `RESEARCH_SUMMARY + COPYWRITING` (из материалов сделать тексты)
   - `TECH_DEBUG + SPEC_ENGINEERING` (диагностика + формальный план фикса)
   - `CRO_AUDIT + COPYWRITING` (аудит + новые текстовые гипотезы)
4. Перед выдачей проверь `forbidden` и соответствие `quality_bar` для каждого выбранного модуля.
